# QA 评审 Rubric

## 目的

当你评估页面主流程是否可靠、状态是否完整、错误是否可恢复、反馈是否足够明确时，使用这份 rubric。

## 全局原则

- 先检查主流程，再看边缘功能
- 要验证状态变化，不只看 happy path
- 技术问题必须落到用户影响上
- 阻塞问题优先于偶发小告警

## 核心问题

- 用户能否识别主操作
- 用户能否完成主任务
- 异步处理中页面是否告诉用户发生了什么
- 出错后页面是否给出可执行的恢复路径
- 空状态和错误状态是否告诉用户下一步怎么做

## 重点维度

### 主流程完整性

重点看：
- 用户是否能找到起点
- 从开始到完成的路径是否短而清楚
- 主操作执行后是否进入正确的下一状态

扣分点：
- 按钮点击无效
- 交互后无任何变化
- 提交陷入循环
- 完成状态不明确

### 状态完整性

检查是否存在：
- loading
- empty
- error
- success
- disabled
- validation feedback
- skeleton 或渐进式内容呈现

扣分点：
- 空白区域替代了明确状态
- 失败后没有恢复提示
- 页面无法执行动作时控件仍然可点

### 交互反馈

重点看：
- 点击、提交、切换、筛选后是否有视觉或内容反馈
- 校验反馈是否及时且明确
- 状态变化是否容易理解

扣分点：
- 用户不知道操作是否生效
- 成功和失败都没有反馈

### Console 与 Network 的用户相关性

仅有 console 噪音，不必自动判为严重问题。

需要升级处理的情况：
- console 错误破坏了主流程
- network 失败让页面停住或误导用户
- 核心数据失败时没有重试或兜底状态

## BLOCKER

出现以下情况视为 blocker：
- 主任务无法完成
- 页面在核心交互中崩溃或不可用
- 必要数据始终无法加载且没有可用兜底
- 校验阻止继续，但没有可执行的指引

## 输出要求

每个重要问题都应说明：
- 用户想做什么
- 实际发生了什么
- 为什么这会影响任务完成
- 支持这个判断的证据是什么
- 主 agent 应该如何修
