# 无障碍与性能 Rubric

## 目的

当你评估页面的基础无障碍、语义、键盘可达性、可读性，以及用户可感知的性能和运行健康度时，使用这份 rubric。

## 全局原则

- 核心操作必须在无鼠标条件下可达
- 语义与 label 在影响理解和导航时必须被重视
- 焦点可见是基础能力，不是装饰
- 性能问题必须映射到用户可感知的成本

## 无障碍检查

### 语义

重点看：
- 是否有合理的 landmark 和 heading
- 表单控件是否有 label
- 纯图标控件是否有可访问名称

### 键盘可达性

重点看：
- 主流程能否通过键盘完成
- 焦点顺序是否合理
- dialog、drawer、menu 是否存在错误的焦点陷阱

### 焦点可见性

重点看：
- 可交互控件是否存在可见焦点状态
- 核心操作是否出现“焦点在但用户看不见”

### 可读性与对比度

重点看：
- 常规缩放下文本是否可读
- 前景色与背景色是否足够清晰
- 颜色是否被作为唯一状态信号

## 性能卫生检查

重点看：
- 明显过长的 loading 且缺乏解释
- 反复闪烁或布局抖动
- 大量资源失败
- Lighthouse 中确实影响体感速度或稳定性的信号

不要过度放大小幅 benchmark 波动，优先关注用户真能感受到的问题。

## BLOCKER

出现以下情况视为 blocker：
- 核心操作无法通过键盘触达
- modal 或 menu 造成严重焦点问题，用户无法完成主流程
- 关键内容对比度过低或不可读
- 运行时失败导致核心内容不可用

## 降级证据策略

如果缺少 accessibility 或 Lighthouse 工具：
- 尽量基于 DOM、截图和交互证据做最佳努力评审
- 明确降低置信度
- 不要假装完成了完整的无障碍或性能覆盖
