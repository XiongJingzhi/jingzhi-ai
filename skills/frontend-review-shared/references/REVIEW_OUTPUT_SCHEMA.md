# 评审输出结构

## Critic 输出

每个 critic 都应按当前对话语言返回以下结构：

### 1. 页面目标判断
- 页面类型
- 你理解的页面目标
- 目标清晰度：`清晰 | 一般 | 不清晰`

### 2. 领域总结
- 本领域总分，或所负责维度的分数
- 该领域结论：`通过 | 有条件通过 | 不通过`
- 置信度：`高 | 中 | 低`

### 3. 问题列表

每个问题都应包含：
- 严重级别：`blocker | high | medium | low`
- 标题
- 为什么重要
- 证据
- 是观察还是推测
- 修复方向

### 4. 最优先修复项

最多 5 条，按影响排序。

### 5. 给主 agent 的修复任务单

使用以下结构：
- 任务目标
- 需要修改的区域
- 必须达成的结果
- 不要做的事
- 验收检查

## Judge 输出

judge 应返回：
- 最终页面目标判断
- 最终总分 `/ 100`
- 最终结论：`通过 | 有条件通过 | 不通过`
- blocker 列表
- 合并后的维度分数
- score gate 结果
- quality gate 结果
- `should_continue_iteration`
- top 5 修复项
- 给主 agent 的最终修复任务单

## 证据规则

每个重要结论都应引用一个或多个证据：
- 截图
- DOM
- 交互结果
- console logs
- network summary
- accessibility summary
- Lighthouse summary

如果证据缺失，必须明确说这是推测。
